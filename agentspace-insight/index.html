分析框架

为什么用 UX 五要素
来看这轮范式迁移

Jesse James Garrett 的五要素模型（战略→范围→结构→框架→表现）最初是为 Web 产品设计的。把它用来分析 Agent 工具，恰恰能照出一个核心矛盾：这些产品在表现层疯狂创新，却在战略层和结构层做出了截然不同的底层赌注——而这些底层差异，才是 3 年后格局分化的真正预言。

01
战略层 Strategy
产品目标与用户需要。分析各产品对「谁是用户」「用户的核心任务是什么」的不同定义，以及这些定义如何决定了其他四层的所有选择。
02
范围层 Scope
功能规格与内容需求。分析哪些功能被选入、哪些被主动排除，以及「功能边界」本身作为战略信号的含义。
03
结构层 Structure
交互设计与信息架构。分析任务流、信息组织方式、人机协作的控制权分配，以及「中间过程可见性」的不同设计哲学。
04
框架层 Skeleton
界面设计、导航设计与信息设计。分析布局模式、模式切换机制、左右面板的功能定义，以及这些选择背后的设计假设。
05
表现层 Surface
感知设计。分析视觉语言、动效系统、品牌一致性，以及「信任感」是如何通过视觉设计来建立或破坏的——这在 Agent 产品里尤为关键。
第一层

战略层：
谁的 Agent，为谁而生

战略层是一切的起点。五款产品面向的用户定义不同，对「成功」的定义不同，由此产生的功能选择和设计决策也完全不同。这一层的分歧，是真正的竞争分水岭。

01
战略层
「我们要做这个产品，是为了达成什么目标？用户希望从中得到什么？」—— Garrett
核心追问：这款产品的用户是谁？他们的核心任务是什么？产品的成功意味着什么？
维度 Claude + Cowork Cursor 3 Codex TRAE SOLO WorkBuddy
核心用户定义 知识工作者（含非技术）+ 开发者双轨 专业开发者；agent 用户已 2:1 超 Tab 用户 高级开发者；云端异步执行场景 研发团队全链路：PM/设计/数据/工程师 非技术职场人员：HR/运营/销售/财务
产品核心目标 让 Agent 能力普惠到所有知识工作场景 让工程师管理「一支 Agent 军队」，不只是用一个 后台异步执行长程编码任务；最小人工干预 打通产品研发上下游；一个工具覆盖所有角色 让非技术人员「一句话指令」完成复杂办公任务
对「Agent」的定义 可信的长程任务执行者 + 知识协作伙伴 可并行调度的软件工程执行单元 后台自主运行的云端代码机器人 跨角色协作的全能产品助手 本地可信的自动化操作员
成功的核心度量 任务完成率 + Cowork/Code 双模采纳 agent 并行任务数 + 开发者日活留存 长程任务自主完成率（最小人工干预） MTC 场景渗透率（非开发者用户占比） 非技术员工任务完成数 + 企业内部采纳
商业赌注 AI 工具的入口从 search 转向 agent OS 「代码开发」赛道的专业工具护城河 开发者不需要 IDE，只需要意图输入框 研发协作的水平整合：覆盖整个产品周期 企业级 AI 办公的本土化 + 安全合规卡位
战略层的核心分歧：「谁的 Agent」决定了「什么叫做完成任务」
Codex 认为成功是「人最少参与」；Cursor 认为成功是「人管理得更多 agent」；WorkBuddy 认为成功是「非技术人员也能用」。这三个定义直接导致了信息架构、验收机制、安全设计的三条完全不同的路径。产品团队在做设计决策前，必须先回答「我们的战略层更像哪个？」
这一轮产品竞争的本质，不是谁的 AI 更强，而是谁对「人机协作比例」的押注更接近用户真实需要。
第二层

范围层：
什么功能被选入，什么被主动排除

功能边界本身是战略声明。一个产品选择不做的功能，往往比它选择做的功能更能说明它对用户的判断。五款产品的范围层呈现出三种完全不同的取舍哲学。

02
范围层
「产品应该包含哪些功能和内容？哪些不应该包含？」—— Garrett
核心追问：取舍的逻辑是什么？功能的边界在哪里划定？谁来决定优先级？
取舍哲学 A
「全栈覆盖」策略
Claude · TRAE SOLO
从编码到办公，从对话到执行，从单人到团队——横向拓展功能边界，以场景覆盖宽度为护城河。代价是产品定位模糊，每个场景都不是最深的。
取舍哲学 B
「专业深挖」策略
Cursor 3 · Codex
只服务开发者，只做编码场景，功能深度优先于宽度。代价是用户群体相对窄，但在核心场景的体验护城河更深。
取舍哲学 C
「垂直切入」策略
WorkBuddy
专注非技术用户的办公场景，以安全合规和本地化为差异点。代价是天花板相对明确，但企业级渗透路径清晰。
功能维度 Claude Cursor 3 Codex TRAE SOLO WorkBuddy
编码执行 深度 Code + Cowork 核心场景 全面 核心场景 专精 Code 模式 不支持
非编码知识工作 深度 Cowork 不支持 不支持 深度 MTC 模式 核心场景
多 Agent 并行 Managed Agents 最深 multi-repo 并行 云端并行 并行独立 context 子任务分配
IDE / 编辑器 保留 IDE 模式 IDE + Agents Window 并存 主动放弃 SOLO 独立于 IDE 无
本地安全 / 沙箱 allowRead 权限 自托管云 agent 云端为主 本地 + 云双版本 Docker/E2B 最深
企业协作 / 权限 RBAC + 团队分析 团队 marketplace 弱 工作空间隔离 企微集成 + 审计
主动排除的功能 无明显主动放弃 非编码办公场景 所有 UI 界面 深度 IDE 功能 编码 / 开发场景
「主动排除」的勇气决定了产品的专注度
Codex 是五款中功能边界最清晰的——主动放弃了 IDE 界面、本地执行、非编码场景。这种「砍功能的勇气」换来了极致简单的交互模型，但也意味着用户迁移成本更低、竞争护城河更薄。反观 TRAE SOLO 同时支持 Code/MTC 的全场景覆盖，横向宽度换来了更大的市场空间，但也带来了「到底在做什么」的产品定位挑战。
第三层

结构层：
任务怎么流动，控制权在谁手里

结构层是 Agent 产品设计中最具决定性的一层。核心问题不是「功能是否完整」，而是「人和 Agent 之间的控制权如何分配」——以及当 Agent 出错时，人如何介入、如何回滚、如何建立信任。

03
结构层
「功能与内容如何相互关联，如何组织？用户如何在产品中移动？」—— Garrett
核心追问：任务流的控制权在哪里？人介入的节点如何设计？错误和意外如何处理？
结构维度 Claude Cursor 3 Codex TRAE SOLO WorkBuddy
信息架构核心隐喻 「任务/对话」并列 + 项目工作空间 「Agent 调度台」——agent 是一等公民，而非对话 「意图输入框」——最小化界面隐喻 「模式化工作台」——Code/MTC 两套并行 IA 「桌面工作台」——文件夹授权 + 技能包执行
任务创建流 对话式输入 → Cowork 执行 → 结果审阅 意图输入 → agent 拆解 → 并行执行 → diff 审查 意图输入 → 后台执行 → 结果通知 意图输入 → 工作空间绑定 → 产物输出 一句话指令 → 技能包匹配 → 本地执行 → 交付
人介入节点设计 Monitor tool 实时响应 + autofix-pr Design Mode 精确反馈 + diff 审查 + /best-of-n 比较 执行后 diff 审查（介入点靠后） 产物可审阅（右侧动态视图） 高危命令拦截 + 沙箱隔离（系统级防护）
错误 / 回滚机制 session recap + resume 恢复 worktree 隔离 + git 集成回滚 弱（依赖 git 外部工具） 弱（产物层面审阅） 沙箱隔离 + 高危拦截（预防为主）
上下文管理架构 1M token 长上下文 + 跨会话 memory multi-repo 上下文 + worktree 隔离 repository 绑定上下文 工作空间持久上下文 + 各 agent 独立 context 授权文件夹 + 技能包 context
并发任务管理 Managed Agents + 多 tab 对话 Agents Window + Agent Tabs 网格视图 后台任务队列 多 agent 并行，各自独立 model + context 子任务动态分配
控制权分配哲学 「可信委托」——人委托，可干预 「主动编排」——人主导调度，agent 执行 「充分授权」——人信任 agent 自主完成 「协作陪伴」——人随时可见可介入 「受控自动化」——系统级安全边界保底
「人介入节点」的设计位置，是结构层最关键的架构决策
Codex 把介入节点放在任务完成后（后验）；Cursor 3 把介入节点分散在执行过程中（Design Mode 实时反馈、diff 审查）；WorkBuddy 把介入节点前置到系统层（高危命令预防性拦截）。这三种设计对应了三种信任哲学：「信任后验收」「信任过程监控」「信任系统边界」。哪种更好？取决于用户对 agent 的信任基线，以及任务的可逆程度。
「上下文可见性」是结构层被严重低估的设计问题
所有产品都在解决「agent 能做什么」的问题，但几乎没有产品系统性地解决「agent 此刻知道什么、不知道什么」的可见性问题。1M token 的上下文窗口让上下文更大，但不等于让用户更清楚。这个设计空白，是下一阶段结构层创新的最大机会点。
第四层

框架层：
布局如何映射心智模型

框架层是战略层的视觉化兑现——界面布局不只是美学选择，它是对「用户在这里最需要什么信息」的核心判断。五款产品在布局设计上呈现出高度收敛的趋势，同时也保留了各自的关键差异。

04
框架层
「界面中各元素如何排列？如何导航？信息如何呈现？」—— Garrett
核心追问：布局选择背后的假设是什么？哪些信息被优先展示？模式切换的心智成本有多高？
框架维度 Claude Cursor 3 Codex TRAE SOLO WorkBuddy
整体布局模式 左栏任务记录 + 中央对话/执行 + 右侧产物 左侧 Agents sidebar + 中央 Agents Window 单栏极简：意图输入 + 结果输出 左侧任务记录 + 中央对话 + 右侧动态视图 对话流 + 文件夹视图 + 工具执行区
左侧面板定义 任务历史 + 项目分组 + 模式切换 所有 agent 状态（跨来源聚合） 无左侧面板 工作空间 + 任务记录 + 模式切换入口 任务历史 + 文件夹权限管理
中间区域定义 对话流 + task monitor（展开/收起） agent 对话 + 执行进度 纯对话/结果视图 对话流（Team Leader 主导） 对话流 + 步骤分解
右侧面板定义 Browser/Terminal/Diff/File 产物 tabs 内嵌浏览器 + 代码编辑器 无（或最小化） 动态视图（产物实时展示） 执行结果文件区
模式切换机制 顶部 tab 切换（Chat / Cowork / Code） 顶部 pill：IDE ↔ Agents Window（可并存） 无需切换（单一模式） 左上角 Code / MTC 切换 无显式模式切换
收起/展开设计 左右面板独立收起；task monitor 独立控制 Agents Window 全屏或分屏 N/A 右侧动态视图可展开 基础展开收起
输入区域设计 @ 符号 + 模式选择 + agent/agent team 切换 / 命令 + @ 文件 + 模型选择器 纯文本意图输入 角色配置 + 模型选择 + 语音输入 自然语言指令框
「三栏布局」正在成为 Agent 工具的新标准框架
Claude、TRAE SOLO、Cursor 3 均在向「左侧任务历史 + 中央对话流 + 右侧产物/工具区」三栏格局收敛。这不是巧合，而是 agent 交互范式的内在逻辑使然：任务历史需要持久可见（左）、执行过程需要实时展示（中）、产物需要随时查验（右）。Codex 的单栏设计是一个刻意的逆势选择，背后是「信任 agent，不需要这些脚手架」的哲学。
模式切换的「摩擦感」是框架层最被低估的设计挑战
TRAE SOLO 的 Code/MTC 切换、Claude 的 Chat/Cowork/Code 切换、Cursor 的 IDE/Agents Window 切换——这些模式边界的设计，直接决定了用户切换场景的心智成本。Cursor 允许 IDE 和 Agents Window「同时开启」，是一个刻意降低切换成本的设计选择，代价是界面复杂度升高。如何让「切换即获得新能力」而非「切换即迷失」，是框架层最值得深挖的设计命题。
第五层

表现层：
信任感是如何被视觉建立的

表现层在 Agent 产品中有一个特殊的任务：建立「可信度」。不同于普通 SaaS 产品，Agent 在执行任务时会读取文件、运行代码、修改内容——视觉语言必须同时传递「能力感」和「可控感」。

05
表现层
「用户最终感知到的产品是什么样子？视觉、听觉、触觉体验如何？」—— Garrett
核心追问：视觉语言如何建立信任？「agent 正在工作」的状态如何感知？品牌一致性是否延伸到 agent 的输出？
表现维度 Claude Cursor 3 Codex TRAE SOLO WorkBuddy
整体视觉风格 柔和米白 + 克制中性；品牌紫色点缀 深色 IDE 传统 + Agent Window 新设计语言 OpenAI 系：极简纯白，几乎无视觉装饰 深灰中性 + 高亮紫色；密度中等 轻量国产 SaaS 风格；功能密度高
「agent 执行中」的状态表达 进度动效 + session recap + Monitor 实时流 Agents Window 实时进度 + diff 流式展示 简单 loading 状态 右侧动态视图实时反馈（最直观） 步骤分解 + 进度条
代码/产物的视觉呈现 语法高亮 diff + 文件树 + Terminal 输出 原生 IDE 代码视图（最精深） 代码 diff 视图 产物预览 + 动态视图 文档/PPT/表格直接预览
错误 / 警告的视觉设计 系统级权限提示 + 高危命令警告 inline 错误提示 + lint 集成 弱（主要依赖文本） 基础错误提示 高危命令拦截弹窗（最显眼）
角色 / 身份感设计 Claude 品牌一致；Cowork 有独立产品感 「编程工具」专业感强；agent 无独立角色感 极简；无角色感 多 agent 角色卡片（有差异化视觉） 技能包有独立 icon（浅层角色感）
信任感建立机制 透明度（context 显示）+ 品牌一致性 专业深度感 + 代码可见性 极简「聪明感」 过程可见性（动态视图） 安全感（沙箱 + 拦截的视觉反馈）
「过程可见性」是 Agent 产品信任感的核心视觉命题
TRAE SOLO 的右侧动态视图是目前五款产品中「过程可见性」最高的设计。WorkBuddy 通过拦截弹窗建立安全感。而 Codex 则是刻意压低可见性——极简 loading 背后是「请信任我」的设计哲学。这三种路径对应了三种截然不同的信任建立策略：「让你看见」「让你有控制权」「让你不必看」。对于 agent 这类高自主性产品，哪种信任策略更适合普通用户？目前行业仍在探索。

对应要素 行业观察 对你们的设计启示 具体行动
战略层 TRAE SOLO 和 WorkBuddy 的「模式分离」背后，是对两类用户心智的主动切割——编码用户和日常办公用户根本是两套交互语言 你们的 Claw/Code 分区方向正确，但需要更清晰地回答「Claw 模式下的成功是什么」——它的产物是文档、分析报告、还是工作流自动化？ 为 Claw 模式定义 3 个核心任务场景 + 成功标准，作为后续功能取舍的决策依据
范围层 Codex 「主动放弃 IDE」的勇气让它的功能边界极清晰；反观全场景产品的定位模糊风险 Agent Space 作为 IDE 内的「第二模式」，需要明确它的边界：哪些任务留在 IDE 模式做，哪些任务只能在 Space 模式做？边界模糊会让用户困惑 画一张「任务分流图」：哪些任务走 IDE 模式，哪些走 Agent 模式，判断标准是什么
结构层 行业最大的结构层空白：「agent 此刻知道什么、不知道什么」的上下文可见性，没有产品系统解决 在你们的 Space 设计中，右侧工具面板底部的「Context 消耗」展示是一个起点，可以进化为「上下文面板」——显式展示 agent 当前的信息边界 设计一个「context scope panel」：当前任务上下文包括哪些文件/内容，用户可手动增减，agent 据此执行
结构层 Cursor 3 的「人介入节点」分散在执行全程（Design Mode + diff + /best-of-n），WorkBuddy 的「高危拦截」是系统级前置防护 你们的 task monitor 目前只做进度展示，需要升级为「介入控制面板」——包含：暂停/继续、修改任务方向、局部回滚三个基础介入能力 在 task monitor 中增加「介入操作区」：pause · redirect · rollback，对应三种介入场景
框架层 「三栏布局」（左任务历史 + 中对话流 + 右产物）已成行业标准框架；Cursor 3 允许 IDE/Agents Window 并存降低切换摩擦 你们的三栏框架方向对；关键是 IDE ↔ Space 的切换体验——切换时不应该「重置状态」，用户的上下文和任务历史应该在两个模式间无缝携带 设计「模式切换时的状态保留规则」：哪些状态跨模式持久，哪些重置，明确用户预期
表现层 TRAE SOLO 的右侧动态视图是过程可见性最高的设计，但所有产品都缺乏「agent 为什么这样做」的决策透明度 右侧产物面板可以升级：不只展示「做了什么」（diff/文件），也展示「为什么这样做」（agent 的决策摘要）——这是表现层建立信任感的最大差异化机会 在产物面板顶部增加「agent 决策摘要」区域：3-5 句话说明这个任务的执行逻辑和关键决策点
一句话总结：行业在「能做什么」上已经趋同，差距将在「如何让人信任」上分开
到 2026 年底，五要素的战略层和范围层将基本趋同——每款产品都会有 Agent、多模式、工作空间。真正分化格局的，是结构层的「控制权设计」和表现层的「信任感建立」——也就是这份报告反复强调的两个问题：人介入节点设计得是否优雅？过程透明度是否让用户敢于放手？这两个问题，是你们 Agent Space 下一阶段设计的核心战场。
