产品概览

三种 Agent Team 的
底层架构差异

表面上，Claude Code Agent Teams、Qoder 专家团、Kimi Agent Swarm 都是「多 agent 并行执行任务」。但在架构层面，三者的 agent 间通信方式、编排来源、角色定义机制截然不同——这些差异不是实现细节，而是产品哲学的体现。

Claude Code
「工程师团队」模型
Squad Collaboration
多个独立 Claude 实例（各有 1M token 上下文）通过 mailbox 系统点对点通信，共享 task list 协作。不依赖中心节点——前端 agent 可以直接告知后端 agent API 变更，无需经过 Team Lead。
P2P 通信 独立 context 去中心化 terminal-first
Qoder 专家团
「项目经理 + 专家」模型
PM-Led Delegation
Team Lead 作为「大脑」统一理解目标、拆解任务、协调全局、质量把关。专家并行工作，但协调权集中在 Lead。用户可随时补充需求，Lead 实时适应变化，无需停止执行。
中心化 Lead 实时可干预 计划先行 IDE 内嵌
Kimi Agent Swarm
「自组织集群」模型
Self-Directed Swarm
无预定义角色，orchestrator 用专项 RL 训练（PARL）学会自主分解任务、动态实例化 frozen sub-agents 并行执行。sub-agent 执行完毕即消亡，结果汇聚给 orchestrator 合并。
PARL 专训 动态实例化 无预设角色 云端集群
同样是「多 agent」，Claude Code 让 agent 像工程师一样互相沟通，Qoder 让 agent 像员工一样听从 PM 指挥，Kimi 让 agent 像集群一样自主分裂执行——三种组织哲学，三套 UX 体系。
第一层 · 战略层

战略层：
为什么要做 Agent Team

三款产品做 Agent Team 的出发点不同，面向的核心用户不同，「成功」的定义也截然不同。战略层的分歧，决定了后四层的所有设计选择。

01
战略层 Strategy
「我们为什么做 Agent Team？这个功能为谁服务？用户核心需求是什么？」
核心追问：Agent Team 是解决「单 agent 上下文不够」的工程问题，还是解决「复杂任务需要专业分工」的协作问题，还是解决「大规模任务需要速度」的效率问题？
Claude Code
战略定位：解决单 agent 上下文溢出 + 工程任务自然分层协作

核心用户是专业开发者，执行跨层（frontend/backend/test）的大型工程任务。Agent Team 的本质是把「一个 agent 撑爆 context」变成「多个 agent 各管一层，互相告知变更」。

成功标准：并行速度提升 + 代码质量（Claude Code Review 内测覆盖率从 16% 升至 54%）
Qoder 专家团
战略定位：从「AI 程序员」进化为「AI 工程交付团队」

核心用户是中高级开发者或技术 PM，提交需求，由专家团完成方案设计→编码→测试→评审的完整交付物。你拿到的不是代码片段，是一份经过规划的工程结果。

成功标准：质量提升（内部测试 +67%）+ 用户「可以不盯着」的信任感
Kimi Agent Swarm
战略定位：让单一模型获得「规模化并行」的能力，速度是第一性能指标

核心用户跨度最大：从开发者到研究人员到内容创作者。Swarm 的价值不在于专业分工，而在于「把需要串行100步的任务压缩为并行10步」。执行时间减少最高 4.5x。

成功标准：执行速度 + 大规模任务完成率（BrowseComp 86.3，领先 GPT-5.4）
战略维度 Claude Code Qoder Kimi
核心问题假设 单 agent context 有限 + 工程任务天然分层，需要专家各司其职并互通 AI 能力需要从「辅助编码」升级为「完整交付」，用户需要信任而非监控 复杂任务的天花板是串行执行速度，并行才能突破
目标用户 专业开发者（Pro/Max/Team）；需要 tmux 等工具配置 中高级开发者、技术 PM；期望完整交付物而非代码片段 开发者 + 研究者 + 内容工作者；最宽泛的目标用户群
产品成熟度赌注 工程严谨性：去中心化 P2P 架构是长期技术赌注 体验完整性：「提需求→拿结果」全流程顺滑是核心护城河 规模优先：300 agents 是技术 benchmark，但实际使用场景仍在探索
开放度 高度开放（subagent 定义、工具权限完全可配置） 黑盒偏高（用户无需了解内部专家分工） 完全黑盒（orchestrator 自主决策，用户不参与角色设计）
战略层洞察
「解决上下文溢出」vs「提升交付质量」vs「压缩执行时间」——三个起点，三条产品路径
Claude Code 的战略起点是工程架构问题（context 隔离、跨层协作）；Qoder 的起点是用户体验问题（交付完整性、信任感）；Kimi 的起点是模型能力边界问题（突破串行执行上限）。这三个起点导致了截然不同的设计优先级——Claude Code 更关注「agent 如何协调」，Qoder 更关注「用户如何验收」，Kimi 更关注「任务如何分解」。
第二层 · 范围层

范围层：
做了什么，刻意不做什么

Agent Team 模式下，功能边界的选择直接暴露了每款产品对「用户应该控制多少」的判断。从角色可定制性到介入机制，三款产品划定了三条不同的功能边界线。

02
范围层 Scope
「在 Agent Team 模式下，包含哪些功能？主动排除了什么？取舍逻辑是什么？」
核心追问：角色定义权交给谁？用户能看到多少中间过程？错误处理机制在哪一层？
功能维度 Claude Code Agent Teams Qoder 专家团 Kimi Agent Swarm
角色定义权 用户完全控制
用户在 prompt 中指定角色（strategist/researcher/reviewer），或通过 .claude/agents/ 配置文件预定义角色；模型也可自主新增 agent		 系统自动分配
Team Lead 根据需求分析自动拉起对应专家，用户无需指定角色；角色类型（架构/编码/测试/QA）由系统内置		 完全黑盒
Orchestrator 自主决定拆分方式和 sub-agent 数量（最多 300），用户无从干预角色创建过程
任务分解可见性 执行前用户可通过 prompt 设计任务结构；执行中可通过 tmux 面板观察各 agent 活动 执行前审阅 Team Lead 生成结构化实施计划，用户可修改后再确认执行 对用户不可见；orchestrator 内部分解，用户只看到最终结果（或 task map 里的 agent 状态）
中途介入机制 可通过 tmux 直接与单个 teammate 交互，无需经过 Team Lead；可实时重定向单个 agent 无需停止会话：随时补充需求、修正方向、变更优先级；Team Lead 实时吸收并重新分配 K2.6 的 Claw Groups 允许人类加入 swarm 协作；但传统 Swarm 模式介入点极少
错误处理机制 各 agent 独立 context 隔离，单 agent 失败不污染全局；用户可手动介入修复单个 agent 高危命令拦截；Team Lead 实时监控专家输出，发现偏差可协调调整；极少数情况需用户确认 Orchestrator 检测到 sub-agent 失败或停滞时自动重新分配任务；K2.6 有自动故障恢复
主动排除的能力 不提供图形化管理界面（纯 terminal）；不自动选择模型分级（社区有需求） 不暴露 agent 内部分工细节（黑盒交付）；不支持 P2P agent 间消息 不支持用户自定义角色；不提供详细的 agent 通信日志
适用任务类型 跨层工程任务（frontend+backend+test）；竞争假设调试；大型代码库重构 中高复杂度开发任务；需要完整交付物（规划→编码→测试→评审） 大规模信息检索和研究；批量内容生成；宽度优先型任务（100 域名 × N 维度）
范围层洞察
「角色定义权」的归属，是三款产品范围层最关键的分叉
Claude Code 把角色定义权完全交给用户（高灵活性，高学习成本）；Qoder 把它收回给 Team Lead（低门槛，低透明度）；Kimi 完全黑盒（零配置，零控制）。这三种选择背后是对用户「能力水平」和「信任意愿」的不同判断——越是面向专业开发者，越倾向于把控制权外露；越是面向普通用户，越倾向于封装成黑盒。
第三层 · 结构层

结构层：
Agent 如何协调，人如何介入

结构层是 Agent Team 设计中最技术性也最关键的一层：信息流动路径、任务分配机制、人的介入节点——这些「看不见的架构」决定了产品在实际使用中的体验质量和可靠性。

结构层最大差异
通信拓扑决定了「协作」的深度上限
Claude Code 的 P2P mailbox 是三款中结构最复杂、能力上限最高的设计——前端 agent 可以直接告知后端 agent「我需要这个 API」，不需要 Team Lead 中转，这才是真正的「工程师团队协作」。Qoder 的中心化 Lead 简化了协调逻辑，代价是 Lead 成为单点瓶颈。Kimi 的单向辐射型只适合「独立子任务 → 结果聚合」型任务，不适合子任务间有深度依赖的工程协作。
第四层 · 框架层

框架层：
Agent Team 的界面如何组织

框架层是战略层的视觉兑现。Agent Team 的状态该怎么展示、任务进度在哪里显示、用户如何看到「正在发生什么」——三款产品在框架层做出了截然不同的选择，对应了三种完全不同的「可见性哲学」。

04
框架层 Skeleton
「Agent Team 模式下，界面如何布局？Agent 状态如何可视化？用户如何导航？」
核心追问：多个 agent 同时运行时，界面如何让用户知道「全局在哪里」「单个 agent 在干什么」「整体进度如何」？
Claude Code
Terminal + tmux 多面板

每个 teammate 一个 tmux 面板，可独立观察；没有图形化主界面。用户通过命令行直接与任意 teammate 交互，或查看 /agents 标签页的 Running/Library 视图。

可见性哲学：「工程师级透明」 — 不做封装，直接把原始执行状态暴露给用户，用户看到的是 agent 真实的输出流。
Qoder 专家团
IDE 内嵌任务列表 + 通知系统

聊天面板底部实时任务列表（待处理/进行中/已完成）；点击单个任务可查看执行进度和过程；需用户确认时通知展示在对话框上方。

可见性哲学：「按需透明」 — 默认给用户一个清洁的进度视图，需要时可下钻查看单个任务详情。
Kimi Agent Swarm
Task Map 侧边栏（K2.6 Swarm Workstation）

侧边栏展示「task map」，每个 sub-agent 是一个活跃进程节点；可点击任意 agent 查看实时执行内容；云端集群，本地为「指挥台」。

可见性哲学：「群体视图」 — 强调规模感（300 个并发节点），但单个 agent 的决策过程不可读。
框架维度 Claude Code Qoder Kimi
整体界面模式 Terminal-first；tmux 多面板；无 GUI；/agents 视图（Running/Library tab） IDE 内嵌；聊天面板 + 底部任务列表 + 通知栏 Web/App 聊天界面 + 侧边 task map；K2.6 Workstation 有专属桌面客户端
Agent 状态展示 各 agent 独立 tmux 面板，原始输出流；/agents Running tab 显示运行中 subagents 底部任务列表（待处理/进行中/完成）；点击下钻查看单任务 Task map 节点图：每个 sub-agent 是一个点，可点击查看当前执行内容
进度可视化 无结构化进度视图；依赖 agent 自己的输出日志 任务颗粒度进度：每个子任务有明确状态；Team Lead 实时对齐 节点数量和活跃状态；K2.6 Claw Groups 中有协作状态视图
下钻查看能力 最强：可直接与任意 teammate 交互，读取其 context，发送消息 中等：可查看单个任务执行进度和过程，但不能直接与专家对话 有限：可点击 task map 中的 agent 节点查看当前执行内容，不可交互
任务历史 / 记录 session 文件持久化；/resume 可恢复大型 session（67% 速度提升） IDE 内聊天历史；专家团任务结果保留在对话流中 标准聊天历史；Swarm 执行详情不作为独立记录保存
输入框设计 命令行 prompt；支持 / 命令（/team-plan, /team-build 等插件） 自然语言输入框；支持执行中补充需求 自然语言输入框；模式切换（Instant/Thinking/Agent/Swarm）
框架层洞察
「可见性」不是越多越好——三款产品代表了三种完全不同的可见性哲学
Claude Code 给你最高的可见性（直接看到每个 agent 的原始输出），代价是认知负载最高；Qoder 给你结构化的进度视图，隐藏细节，降低认知负载；Kimi 给你规模感但不给你决策过程，适合「我只关心结果」的用户。哪种可见性水平是对的？取决于用户对 agent 的信任基线和干预意愿——这正是产品团队需要通过用户研究来校准的核心参数。
第五层 · 表现层

表现层：
如何让用户「感受到」Agent Team 在工作

当 300 个 agent 并行运行时，用户的感知是什么？是秩序感、能量感，还是不安感？表现层决定了 Agent Team 这一抽象概念如何转化为用户可感知的体验——信任感在这一层被建立或摧毁。

05
表现层 Surface
「Agent Team 工作时，用户的视觉体验是什么？如何传递「有序、可信、高效」的感知？」
核心追问：多 agent 并行执行时的视觉语言是什么？进度、速度、秩序感如何通过设计传达？agent 的「角色感」有多强？
表现维度 Claude Code Qoder Kimi
「Agent 工作中」的感知 多 tmux 面板同时滚动输出流；raw terminal 输出；视觉密度极高，「工程师感」强 底部任务列表动态更新；状态标签颜色变化（进行中/完成）；通知出现时有明显提示 Task map 节点闪烁/活跃状态；数字跳动（N agents running）；视觉上的「蜂群感」
角色差异化设计 弱角色感：各 teammate 名字不同（frontend/backend/test）但视觉上无差异 中等角色感：专家有内置角色名（架构师/工程师/测试员/评审员），列表中可区分 无角色感：sub-agent 是匿名执行单元（SA-01/02...），无角色标识
进度感知设计 无结构化进度指示；靠 agent 输出量感知进度 任务状态条（待处理→进行中→已完成）+ 实时任务计数；有明确的完成感 K2.6 数字量化：「N agents running, M steps completed」；规模即进度
品牌一致性 Claude 品牌延伸；terminal 工具感；与 Claude.ai 视觉割裂 Qoder IDE 设计语言一致延伸；团队执行感与品牌「专业工程」定位吻合 Kimi 聊天界面一致；Kimi K2.6 Swarm Workstation 有独立视觉语言
错误 / 等待的表现 terminal 报错原样显示；无视觉柔化处理；需用户自己判断严重性 高危命令拦截时弹窗提示；等待用户确认的通知在对话框上方显著展示 orchestrator 自动重分配失败任务，用户通常不感知单个 agent 的失败
信任感建立机制 透明度（看到一切）+ 专业工具感（terminal 让有经验的开发者安心） 结构化进度 + 计划先行（执行前看到 plan）+ 随时可干预机制 规模数字（300 agents）+ 速度（4.5x）+ 自动容错（orchestrator 兜底）
表现层最关键的设计问题
「Agent 有角色感」vs「Agent 是匿名执行单元」——这个选择影响的不只是视觉
Qoder 给专家贴上「架构师/工程师/测试员/评审员」的角色标签，这不只是 UI 装饰，而是在建立「分工有秩序」的信任感——用户知道「有人在专职做测试」；Kimi 的匿名 sub-agent 则传递「效率机器」的信号，用户信任的是 orchestrator 而非单个 agent；Claude Code 的「frontend/backend/test」命名折中于两者之间。在你们的 Agent Team 设计中，为每个 agent 设计有辨识度的「像素风头像 + 角色名」，是表现层建立信任感的低成本高回报投入。

设计启示

对你们 Agent Team 设计的
六条直接启示

结合 UX 五要素分析，以下启示直接对应三款产品的关键发现，并转化为你们 Agent Space 中 Agent Team 模式的具体设计行动。

战略层 → 结构层
优先回答「什么任务需要 Agent Team，什么只需要单 Agent」
行业共识：Qoder 建议「中高复杂度任务」用专家团，简单任务用单 Agent；Claude Code 的 5-agent 规则是「单 agent context 超 50K 行代码」时考虑 team。你们需要在 Agent/Agent Team 切换的输入框处，给用户清晰的任务复杂度指引，而非让用户盲目选择模式。
范围层 → 框架层
「执行前计划审阅」是 Qoder 最值得借鉴的设计
Qoder 的 Team Lead 在执行前生成结构化实施计划，用户可修改后再确认——这是「人机协作」在范围层最优雅的设计点。你们的 Team Leader 在输出任务分配计划时，可以设计一个「计划确认」步骤，允许用户在执行前调整子任务的优先级和分工，而不是直接开始执行。
结构层
子 Agent 间的通信方式决定了 Agent Team 能解决问题的上限
Claude Code 的 P2P mailbox 让前端 agent 可以直接告知后端 agent API 变更，这是目前行业中协作深度最高的设计。你们如果想让 Agent Team 真正解决跨层依赖问题，需要设计 agent 间的「通知机制」——至少让一个 agent 的关键产出能触发其他相关 agent 的感知。
框架层
Task Monitor 需要做到任务颗粒度，而非 Agent 颗粒度
Qoder 的底部任务列表（待处理/进行中/完成）比 Kimi 的 agent 节点图更实用，因为用户关心的是「任务是否完成」而非「哪个 agent 在跑」。你们的 task monitor 应以「任务/子任务」为展示单位，而非 agent 为单位，并支持点击单个任务下钻查看该任务的执行 agent 和过程。
表现层
Agent 的「角色感」直接影响信任感——像素风头像不只是装饰
三款产品中，Qoder 的「架构师/工程师/测试员/评审员」角色标签最有效地传递了「分工有秩序」的信任感。你们已经设计了像素风 Agent 头像（Team Leader/Arch/DB/Backend/Frontend/Test），这是表现层的重要投入——确保每个 agent 的视觉标识在 task monitor、对话流、下钻详情三个场景下保持一致显示。
结构层 → 表现层
「子 Agent 上下文连贯性」是当前行业最大的结构层未解问题
Qoder 的社区反馈明确指出：跨轮次重启专家时上下文不连贯，导致每次都要重新灌入上下文，token 消耗增大且体验下降。这是行业级的设计难题。你们在设计 Agent Team 时，需要明确「同一任务中，同一个子 Agent 的会话如何保持连续」——这不只是技术问题，也是产品层面需要做出承诺的体验保证。
最终结论
你们的 Agent Team 设计机会：把 Qoder 的「体验完整性」和 Claude Code 的「协作深度」结合起来
当前行业没有一款产品同时做好了「框架层的信息架构」（Qoder 领先）和「结构层的协作机制」（Claude Code 领先）。你们在做 IDE 内嵌的 Agent Space，有机会做到：执行前 Team Leader 生成可审阅的计划（学 Qoder）+ 各 agent 有清晰角色标识（学 Qoder）+ 支持关键产出跨 agent 通知（学 Claude Code）+ 任务粒度的 task monitor（超越两者）+ 产物可见且可验收（你们已有基础）。这个组合，是行业里目前没有人完整做出来的 Agent Team 体验。
